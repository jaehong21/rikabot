# 019 — CLI Lifecycle + Embedded Web Assets

## Context

Rikabot currently starts the HTTP gateway directly from `main.rs` with no CLI surface.

User requirement for this phase:

1. Convert runtime entrypoint to a `clap`-based CLI.
2. Provide standard CLI UX (`--help`, `--version`).
3. Add lifecycle commands:
   - `start`
   - `restart`
   - `stop`
   - `status`
4. Ensure web frontend assets are packaged into the backend binary at build time and served as static files by the gateway.

## Goals

1. Introduce explicit CLI command model without removing existing server behavior.
2. Support background process lifecycle management for the server.
3. Keep gateway WebSocket/API behavior unchanged.
4. Make backend binary self-contained for static frontend serving.

## Non-Goals

1. Full OS service integration (`launchd`, `systemd`, Windows services).
2. Multi-instance orchestration.
3. Adding advanced admin commands beyond requested lifecycle set.
4. Frontend redesign or route behavior changes.

## Scope

### In scope

1. `clap` integration in backend binary.
2. Command handlers for `start/restart/stop/status`.
3. Internal foreground server command path used by lifecycle commands.
4. Runtime PID/state file management in workspace.
5. Build-time web asset preparation and compile-time embedding.
6. Gateway static fallback migration from disk `ServeDir` to embedded asset handler.

### Out of scope

1. Container/process supervisor setup.
2. Distributed process control.
3. New REST API for lifecycle control.

## Functional Requirements

### 1) CLI contract

1. Binary exposes `--help` and `--version` via `clap`.
2. Commands:
   - `start`: start server process (default daemon/background mode).
   - `restart`: stop running daemon (if any), then start again.
   - `stop`: stop running daemon.
   - `status`: report daemon state.
3. Include hidden/internal `serve` command for foreground server execution, used by `start` and local/manual runs.
4. Support optional global `--config <path>` override.

### 2) Lifecycle behavior

1. `start`:
   - If already running, return clear “already running” status.
   - If stale PID/state file exists, self-heal by replacing it.
2. `stop`:
   - If running, send termination signal and wait for exit with timeout.
   - If not running, return clean “already stopped” status.
3. `restart`:
   - Equivalent to `stop` + `start` with same config.
4. `status`:
   - Show running/stopped.
   - Include PID and URL when available.
   - Optionally include health endpoint reachability.

### 3) State management

1. Persist daemon runtime state in workspace (e.g., `<workspace>/runtime/server.json`).
2. Record at minimum:
   - PID
   - host
   - port
   - config path used
3. Commands use this state for lifecycle actions and stale-state cleanup.

### 4) Embedded static assets

1. Backend no longer depends on runtime filesystem `web/dist` serving.
2. Web build artifacts are embedded in binary at compile time.
3. Gateway fallback serves:
   - direct asset path when file exists
   - `index.html` for SPA routes
   - `404` for missing file-like asset paths
4. Keep existing `/health` and `/ws` routes unchanged.

### 5) Build pipeline

1. Add backend build step that ensures `web/dist` exists and is up to date before compilation.
2. Prefer local JS package manager in this order:
   - `bun`
   - `npm`
   - `pnpm`
   - `yarn`
3. Fail build with clear error if assets cannot be built and no valid `web/dist` is available.

## Implementation Plan

### Phase 1: CLI and server split

1. Add `clap` parser types and command dispatch in `main.rs`.
2. Extract current server startup into reusable `run_server(config_path)` function.
3. Introduce internal `serve` command for foreground execution.

### Phase 2: Lifecycle commands

1. Add runtime state file read/write helpers.
2. Implement process checks + signal-based stop flow.
3. Implement `start/restart/status` command handlers.

### Phase 3: Embedded frontend serving

1. Add asset embedding crate integration.
2. Replace `ServeDir` fallback with embedded handler in gateway.
3. Preserve SPA fallback behavior.

### Phase 4: Build integration and verification

1. Add `build.rs` to build/check web assets.
2. Verify `cargo check` on changed crate.
3. Smoke test CLI help + lifecycle command parsing.

## File-by-File Changes

1. `Cargo.toml`
   - add `clap` and web embedding dependencies
2. `build.rs` (new)
   - ensure `web/dist` is available/current
3. `src/main.rs`
   - CLI parser + command dispatch + lifecycle handlers
4. `src/gateway/mod.rs`
   - embedded static asset fallback handler
5. `docs/prd/0019-cli.md` (new)
   - this design document

## Acceptance Criteria

1. `cargo run -- --help` shows lifecycle commands and options.
2. `cargo run -- --version` returns crate version.
3. `cargo run -- start` starts daemon and writes runtime state file.
4. `cargo run -- status` reports running daemon with PID and URL.
5. `cargo run -- stop` stops running daemon and clears stale runtime state.
6. `cargo run -- restart` successfully cycles daemon state.
7. Built binary serves frontend without requiring runtime `web/dist` on disk.
8. `/health` and `/ws` remain functional.

## Risks and Mitigations

1. Risk: stale PID file points to unrelated process.
   - Mitigation: add health check and stale-state cleanup path.
2. Risk: platform-specific stop behavior differs.
   - Mitigation: use conservative signal approach and clear timeout/error messaging.
3. Risk: missing JS toolchain in build environment.
   - Mitigation: allow prebuilt `web/dist`; only fail when assets are unavailable.
4. Risk: SPA fallback may hide missing real asset bugs.
   - Mitigation: return `404` for paths that look like file requests.

## Open Questions

1. Should `start` default to foreground or background mode for this repo’s expected workflow?
2. Do we want a `logs` command in a follow-up for daemon output visibility?
3. Should runtime state be moved from workspace path to OS-specific data directory in a later phase?
