# Rikabot Prototype Plan

## Context

Build a personal AI assistant ("rikabot") in Rust, inspired by the nanobot/zeroclaw architecture. The prototype provides a web-based chat UI with an agent loop that can call LLM providers and execute tools. Memory and skills are deferred to future iterations.

## Architecture Overview

```
                        ┌─────────────────────────────────────────────┐
                        │              Gateway (src/gateway/)         │
Browser ──WebSocket──>  │  Axum HTTP server: routes, WS, static files │
                        └──────────────────┬──────────────────────────┘
                                           │
                        ┌──────────────────▼──────────────────────────┐
                        │              Agent Loop (src/agent/)        │
                        │  message -> LLM -> tools -> response cycle  │
                        └────────┬─────────────────────┬──────────────┘
                                 │                     │
                    ┌────────────▼────────┐  ┌────────▼──────────┐
                    │ Provider (src/      │  │ Tool Registry     │
                    │   providers/)       │  │ (src/tools/)      │
                    │ OpenRouter, Codex   │  │ ShellTool         │
                    └─────────────────────┘  └───────────────────┘

Future:  src/channels/  (Telegram, Discord, Slack, ...)
```

**Gateway vs Channels** (following zeroclaw's architecture):

- **Gateway** (`src/gateway/`) — The HTTP facade. An Axum web server responsible for REST API endpoints, WebSocket agent chat, static file serving (the web UI), and health checks. It is the single HTTP entry point for browsers and programmatic API clients.
- **Channels** (`src/channels/`, future) — Long-running async listeners for messaging platforms (Telegram, Discord, Slack, etc.). Each channel connects to a platform's native API, listens for messages, routes them through the agent loop, and sends replies back. Channels are push-based and run as separate tokio tasks alongside the gateway.

Both gateway and channels feed messages into the same agent loop, but through different transports.

## Core Types & Traits

### Provider Trait (`src/providers/mod.rs`)

Follow zeroclaw's pattern from `/Users/jetty/Desktop/open-source/zeroclaw/src/providers/traits.rs`:

```rust
// Reuse zeroclaw's simple ChatMessage (role + content only)
pub struct ChatMessage { pub role: String, pub content: String }

// Tool call from LLM response
pub struct ToolCall { pub id: String, pub name: String, pub arguments: String }

pub struct ChatResponse {
    pub text: Option<String>,
    pub tool_calls: Vec<ToolCall>,
    pub usage: Option<TokenUsage>,
}

pub struct ToolSpec { pub name: String, pub description: String, pub parameters: serde_json::Value }

#[async_trait]
pub trait Provider: Send + Sync {
    fn supports_native_tools(&self) -> bool;
    async fn chat(&self, messages: &[ChatMessage], tools: Option<&[ToolSpec]>, model: &str, temperature: f64) -> Result<ChatResponse>;
}
```

**Key design choice**: zeroclaw encodes tool_calls and tool_results as JSON strings inside `ChatMessage.content` for assistant/tool role messages. We follow this same pattern to keep `ChatMessage` simple. Each provider's `convert_messages()` method parses these JSON-encoded payloads back into native API format.

```rust
    fn parameters_schema(&self) -> serde_json::Value;
    async fn execute(&self, args: serde_json::Value) -> Result<ToolResult>;
    fn spec(&self) -> ToolSpec { /* default impl */ }
}

pub struct ToolRegistry { tools: Vec<Box<dyn Tool>> }
// Methods: register(), specs(), execute(name, args)
```

### Agent Loop (`src/agent/mod.rs`)

Iterative loop following nanobot's pattern:

```
1. Build messages: [system_prompt, ...history, user_message]
2. Loop (max 20 iterations):
   a. response = provider.chat(messages, tools, model, temp)
   b. If response.has_tool_calls():
      - Append assistant message (with tool_calls encoded as JSON in content)
      - For each tool_call: execute, append tool result message, send AgentEvent to WebSocket
   c. Else: send AgentEvent::Done with final text, break
```

Agent events sent to WebSocket via `mpsc::unbounded_channel<AgentEvent>`:

- `Chunk { content }` - text from assistant alongside tool calls
- `ToolCallStart { name, args }` - tool execution starting
- `ToolCallResult { name, output, success }` - tool finished
- `Done { full_response }` - final answer
- `Error { message }` - error occurred

### Conversation History

Within a single WebSocket connection, maintain a `Vec<ChatMessage>` so the user can have multi-turn conversations. No persistence across connections (future work).

## Provider Implementations

### OpenRouter (`src/providers/openrouter.rs`)

Reference: `/Users/jetty/Desktop/open-source/zeroclaw/src/providers/openrouter.rs`

- API: `POST https://openrouter.ai/api/v1/chat/completions` (OpenAI-compatible)
- Auth: `Authorization: Bearer <api_key>`
- Native tool calling: **yes** (`tool_choice: "auto"`)
- Message conversion: parse JSON-encoded tool_calls/tool_results from ChatMessage.content into native format (NativeMessage with tool_calls array, tool role with tool_call_id)
- Response parsing: extract tool_calls from `choices[0].message.tool_calls`

### OpenAI Codex (`src/providers/openai_codex.rs`)

Reference: `/Users/jetty/Desktop/open-source/zeroclaw/src/providers/openai_codex.rs`

- API: `POST https://chatgpt.com/backend-api/codex/responses` (Responses API)
- Auth: `Authorization: Bearer <access_token>`, `chatgpt-account-id: <account_id>` headers
- Native tool calling: **no** - use prompt-guided fallback (inject tool instructions into system prompt as text)
- SSE streaming response - accumulate `response.output_text.delta` events
- Message format: convert to `ResponsesInput` items with `input_text`/`output_text` content types
- For MVP: accept access_token + account_id from config directly (no OAuth flow)

## Gateway (`src/gateway/`)

The gateway is the HTTP facade for rikabot — an Axum web server that handles all browser/API interactions. Follows zeroclaw's `src/gateway/` pattern.

**Responsibilities:**

1. **Static file serving** — Serve the web UI from `web/` directory (embedded via `include_str!` for prototype; migrate to `rust-embed` later for full SPA)
2. **WebSocket agent chat** (`/ws`) — Real-time bidirectional communication for the agent loop
3. **Health endpoint** (`/health`) — Always public, no auth
4. **Route registration** — Central place for all HTTP routes + middleware

### AppState (`src/gateway/mod.rs`)

```rust
pub struct AppState {
    pub agent: Arc<Agent>,
}
```

Minimal for prototype. Future: `Arc<Mutex<Config>>`, `broadcast::Sender` for SSE events, rate limiter, auth pairing.

### Routes

```
GET  /        → serve web/index.html (SPA entry point)
GET  /health  → "ok"
GET  /ws      → WebSocket upgrade for agent chat
```

### WebSocket Handler (`src/gateway/ws.rs`)

Protocol:

```
Client -> Server: {"type":"message","content":"Hello"}
Server -> Client: {"type":"tool_call","name":"shell","args":{...}}
Server -> Client: {"type":"tool_result","name":"shell","output":"...","success":true}
Server -> Client: {"type":"done","full_response":"Here are the files..."}
Server -> Client: {"type":"error","message":"something went wrong"}
```

Per-connection conversation history maintained in the handler loop.

### Frontend (`web/index.html`)

Separate `web/` directory at project root (following zeroclaw). For prototype, a single self-contained HTML file (~300 lines) embedded via `include_str!("../../web/index.html")`. No build step, no npm.

- Chat bubble layout (user messages right, assistant left)
- Collapsible tool call/result blocks
- WebSocket connection with auto-reconnect
- Minimal dark theme
- Input box with send button + Enter key support

Future: grow into full SPA with `web/src/`, `web/dist/`, Vite build, and `rust-embed` for serving.

## Implementation Order

1. **Scaffold** - Cargo.toml, main.rs skeleton, config.rs
2. **Provider types** - mod.rs with trait, ChatMessage, ChatResponse, ToolSpec, ToolCall
3. **Tool types** - mod.rs with Tool trait, ToolRegistry, ToolResult
4. **ShellTool** - shell.rs implementation with timeout + output capture
5. **OpenRouter provider** - full implementation with native tool calling
6. **Agent loop** - iterative LLM -> tool -> LLM cycle with AgentEvent channel
7. **Gateway + Frontend** - axum router, WebSocket handler, web/index.html
8. **OpenAI Codex provider** - Responses API with SSE parsing, prompt-guided tools
9. **Config polish** - env var overrides, config.toml.example
