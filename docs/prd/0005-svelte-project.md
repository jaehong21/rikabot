# Web Frontend Migration to Svelte 5 (Bun)

## Context

The current frontend is a single static file at `web/index.html` with inline CSS/JS. It works, but it is hard to extend safely because UI, state management, and transport logic are all mixed together in one document.

We need to migrate the frontend to a Svelte 5 project so future UI features can be added with component-based structure, typed modules, and a modern build pipeline.

## Goals

1. Replace static `web/index.html` with a Svelte 5 app scaffold under `web/`.
2. Use `bun` as package manager and script runner (`package.json` under `web/`).
3. Preserve current chat behavior:
   - WebSocket connection to `/ws`
   - Streaming assistant chunks
   - Tool call start/result blocks
   - Reconnect behavior and loading states
4. Deliver a clean, modern interface direction inspired by ChatGPT / ElevenLabs:
   - restrained palette
   - clear hierarchy
   - polished message/input surfaces
5. Update Rust gateway static serving for built frontend assets from `web/dist`.

## Non-Goals

- Introducing auth, persistence, or multi-page frontend routing.
- Reworking agent protocol/event schema.
- Adding external UI component libraries.

## Architecture Changes

### Frontend

- Build tool: Vite
- Framework: Svelte 5
- Language: TypeScript
- Files:
  - `web/index.html` (Vite entry)
  - `web/src/main.ts`
  - `web/src/App.svelte`
  - `web/src/app.css`
  - `web/package.json`, `web/tsconfig*.json`, `web/vite.config.ts`, `web/svelte.config.js`

### Backend (Axum)

- Replace compile-time `include_str!("../../web/index.html")` serving with static file serving from `web/dist`.
- Serve:
  - `/assets/*` and other built files from `web/dist`
  - `/` (and fallback) as `web/dist/index.html`
- Keep `/ws` and `/health` unchanged.

## Migration Plan

### Phase 1: Scaffold and dependencies

1. Initialize Svelte 5 + Vite project under `web/` using Bun.
2. Add scripts:
   - `bun run dev`
   - `bun run build`
   - `bun run preview`
   - optional typecheck/lint scripts if generated by scaffold

### Phase 2: Port chat client logic

1. Move WebSocket logic and event handling into Svelte component state.
2. Preserve current message semantics:
   - user/assistant message bubbles
   - chunk accumulation
   - tool call blocks with expandable output
   - done/error handling
3. Keep textarea autosize + Enter-to-send behavior.

### Phase 3: UI refresh

1. Implement a clean layout:
   - top bar with connection state
   - scrollable chat pane
   - anchored composer
2. Use subtle elevation, neutral dark surfaces, and focused accents.
3. Keep mobile responsiveness and desktop readability.

### Phase 4: Rust gateway integration

1. Update Axum router to serve built assets from `web/dist`.
2. Add static fallback to `index.html` for SPA behavior.
3. Keep request timeout and existing routes.

## File-by-File Changes

- `web/` (new project files replacing static page structure)
- `src/gateway/mod.rs` (serve dist files instead of embedded HTML)
- `Cargo.toml` (enable `tower-http` filesystem serving feature)

## Verification

1. Frontend:
   - `cd web && bun install`
   - `cd web && bun run build`
2. Backend:
   - `cargo build`
3. Manual runtime:
   - Run server, open `/`
   - Send message, verify streamed response and tool blocks
   - Confirm reconnect overlay/state on socket interruption
4. Visual:
   - Verify layout and spacing on desktop and mobile widths

## Risks and Mitigations

- Risk: Missing `web/dist` in environments that run backend without frontend build.
  - Mitigation: explicit build step in docs/workflow and startup log message if assets missing.
- Risk: Behavior regression while porting monolithic JS to component state.
  - Mitigation: preserve existing event model and manually verify each event type.

## Open Question

Should production binary eventually embed built frontend assets, or is serving `web/dist` from disk acceptable for now?
